# DB 인덱스 (Index)

## 인덱스란?

인덱스는 데이터베이스 테이블의 검색 속도를 향상시키기 위한 자료구조로, 백과사전의 색인과 같습니다. 저장되는 컬럼의 값을 사용하여 항상 정렬된 상태를 유지하는 것이 특징입니다.

### 인덱스의 특징
- 검색(SELECT) 성능을 크게 향상시킴
- 항상 정렬된 상태를 유지
- INSERT, UPDATE, DELETE 시 인덱스 갱신도 필요하므로 쓰기 성능 저하가 발생

---

## B+Tree 인덱스

MySQL InnoDB를 기준으로, B+Tree와 같은 변형 B-Tree 자료구조를 이용해서 인덱스를 구현합니다.

### B-Tree 구조
B-Tree(Balanced-Tree)에서는 크게 3가지 노드가 존재합니다.

- **루트 노드(Root Node)**: 최상위에 하나의 루트 노드
- **리프 노드(Leaf Node)**: 가장 하위 노드, 실제 데이터 레코드를 찾아가기 위한 주소값을 가짐
- **브랜치 노드(Branch Node)**: 루트 노드와 리프 노드 사이에 존재하는 중간 노드

B-Tree 인덱스는 컬럼의 값을 변형하지 않고 인덱스 구조체 내에서 항상 정렬된 상태로 유지합니다.

### InnoDB의 세컨더리 인덱스
InnoDB 스토리지 엔진에서는 세컨더리 인덱스(프라이머리 인덱스를 제외한 모든 인덱스)의 리프 노드에는 레코드의 PK가 저장됩니다. 따라서 세컨더리 인덱스 검색에서는 레코드를 읽기 위해 PK를 가지고 있는 B-Tree를 다시 한번 검색해야 합니다.

## 인덱스 스캔 방식

MySQL에는 크게 세 가지 인덱스 스캔 방식이 있습니다.

### 1. 인덱스 레인지 스캔 (Index Range Scan)
검색할 인덱스 범위가 결정되었을 경우 사용하며 가장 빠릅니다.

동작 과정:
1. 인덱스에서 조건을 만족하는 값이 저장된 시작 리프 노드를 찾습니다. (index seek)
2. 시작 리프 노드부터 필요한 만큼 인덱스를 차례대로 읽습니다. (index scan)
3. 인덱스 키와 레코드 주소를 이용해 저장된 페이지를 가져오고 레코드를 읽어옵니다.

주의사항:
- 레코드를 읽어오는 과정에서 랜덤 IO가 발생할 수 있음
- 읽어야 할 데이터 레코드가 전체의 20~25%를 초과하는 경우에는 풀 테이블 스캔(순차 IO)이 더 효율적일 수 있음

### 2. 인덱스 풀 스캔 (Index Full Scan)
인덱스를 사용하지만 인덱스를 처음부터 끝까지 모두 읽는 방식입니다.

- 인덱스를 ABC 순서로 만들었는데 조건절에 B 혹은 C로 검색하는 경우 사용됨
- 인덱스를 생성한 목적에는 맞지 않지만, 데이터 레코드까지 읽지 않는 경우 풀 테이블 스캔보다는 효율적

### 3. 루스 인덱스 스캔 (Loose Index Scan)
듬성듬성하게 인덱스를 읽는 방식입니다. (인덱스 레인지 스캔과 인덱스 풀 스캔은 `타이트 인덱스 스캔`으로 분류)

- 중간에 필요하지 않은 인덱스 키 값은 무시하고 다음으로 넘어가는 형태
- `GROUP BY`, `MAX()`, `MIN()` 함수에 대해 최적화하는 경우에 사용

## 인덱스 사용 시 주의사항

1. **선택도(Selectivity)를 고려하라**: 카디널리티가 낮은 컬럼(예: 성별, 불리언)에 인덱스를 걸면 효과가 거의 없음
2. **복합 인덱스의 순서**: 복합 인덱스에서는 맨 앞 컬럼을 조건에 포함해야 인덱스를 타는 경우가 많음
3. **인덱스 컬럼에 함수/연산 사용 금지**: `WHERE YEAR(created_at) = 2024`와 같이 인덱스 컬럼에 함수를 적용하면 인덱스를 사용하지 못함
4. **쓰기 성능 저하**: 인덱스가 많을수록 INSERT, UPDATE, DELETE 시 인덱스 갱신 비용이 증가
5. **커버링 인덱스 활용**: 쿼리에서 필요한 컬럼이 모두 인덱스에 포함되어 있으면 레코드를 읽지 않아도 되어 성능이 크게 향상됨
6. **20~25% 규칙**: 조회 범위가 전체 데이터의 20~25%를 초과하면 풀 테이블 스캔이 더 효율적일 수 있음
