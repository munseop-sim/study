# 소프트웨어 아키텍처

## 레이어드 아키텍처 (Layered Architecture)

**레이어드 아키텍처(Layered Architecture)** 란 소프트웨어를 관심사별로 여러 계층으로 나누어 수직적으로 배열한 것을 의미합니다. 여기서 관심사란 유사한 책임들을 의미합니다. 예를 들어, 데이터베이스 접근과 관련된 책임들을 하나의 관심사로 볼 수 있습니다.

### 구성 계층

레이어드 아키텍처의 대표적인 구성에는 3가지 레이어가 존재하는데요. 레이어의 종류와 수는 프로젝트 상황마다 달라질 수 있습니다.

- **표현 계층(Presentation Layer)**: 사용자 입력을 처리하기 위해 존재합니다.
- **도메인 계층(Domain Layer)**: 비즈니스와 관련된 로직을 수행하기 위해 존재합니다.
- **데이터 소스 계층(Data Source Layer)**: 데이터베이스 접근 및 데이터 조작과 관련된 작업을 수행하기 위해 존재합니다.

### 장점

1. **재사용성**: 각 계층이 정의된 기능에 집중하기 때문에, 상위 계층은 하위 계층에 정의된 기능을 필요에 따라 재사용 가능합니다.
2. **유지관리**: 컴포넌트가 논리적으로 분리되어 있기 때문에, 각 계층의 유지 보수와 수정이 용이합니다.
3. **변화에 대한 유연성**: 특정 레이어만 독립적으로 확장하거나 변경할 수 있으며, 시스템 전체를 변경하지 않고도 한 계층의 기술이나 구현을 변경할 수 있습니다.
4. **개발의 단순성**: 각 계층은 다른 계층의 결과에 의존하기 때문에, 개발자는 특정 계층에 집중하며 작업을 수행할 수 있습니다.
5. **테스트 용이성**: 특정 레이어만 별도로 테스트 환경을 구축하여 테스트할 수 있습니다.

### 단점

1. **성능**: 하위 계층을 반복적으로 호출함으로써 발생하는 오버헤드가 성능에 영향을 줄 수 있습니다.
2. **복잡성**: 때로는 실제로 필요하지 않은 레이어를 포함하는 경우가 있을 수 있어 설계와 구현이 복잡해질 수 있습니다.
3. **묶여 있는 결합도**: 한 계층이 다음 계층에 너무 의존하게 되면, 계층 간의 결합도가 증가할 수 있습니다. 이로 인해, 중간 계층에 발생하는 문제가 상위 계층에 영향을 줄 수 있습니다.

### 싱크홀 안티 패턴 (Architecture Sinkhole Anti-Pattern)

일반적으로 레이어드 아키텍처에서 요청은 상위 레이어(표현 계층)에서 중간 레이어를 거쳐 하위 레이어(데이터 소스 계층)로 전달됩니다. 이때, **중간 레이어는 아무 일도 하지 않음에도 불구하고 요청을 무작정 통과시키는 것**을 싱크홀 안티 패턴이라고 합니다.

```java
@Service
public class OrderService {

  private OrderDao orderDao;

  // 아무 일도 수행하지 않음
  public OrderResponse getOrder(Long orderId) {
      return orderDao.getOrderById(orderId);
  }
}
```

- 불필요하게 요청을 전달받고, 다시 전달만 하는 중간 코드를 작성하게 됩니다.
- CPU 및 메모리 자원을 낭비한다는 문제가 있습니다.
- 하지만 이를 무조건 피해야 하는 것은 아닙니다. 상위 레이어가 직접 하위 레이어에 접근하는 방식을 허용하면 일관성이 약해져 추가적인 소통과 문서가 필요할 수 있습니다. 프로젝트와 팀의 상황에 맞게 트레이드오프를 고려하여 팀만의 규칙을 만드는 것이 중요합니다.

## 헥사고날 아키텍처 (Hexagonal Architecture)

헥사고날 아키텍처(또는 포트와 어댑터 아키텍처)는 소프트웨어 설계 패턴 중 하나입니다.

- **내부(Service, 도메인 로직)**: 도메인 로직이 위치하며, `in.port`를 구현하고 `out.port`를 주입받아서 사용합니다.
- **외부(Adapter)**: 외부로부터 도메인 로직에 접근 또는 도메인 로직에서 외부로 접근하는 자원을 의미합니다.
- **인터페이스(Port)**: 도메인 로직과 외부 간의 연결입니다.

### 장점

1. **추상화**: 핵심 비즈니스 로직은 외부 구현 세부사항과 분리됩니다.
2. **테스트 용이성**: 핵심 비즈니스 로직은 고립된 상태에서 쉽게 테스트할 수 있습니다.
3. **유연성**: 다양한 인프라스트럭처 기술을 쉽게 사용하거나 교체할 수 있습니다.
4. **장기적인 관리**: 기술적인 부분과 비즈니스 로직이 분리되어 있어, 애플리케이션의 유지보수가 용이합니다.

### 단점

1. **복잡도**: 처음에는 복잡한 아키텍처가 과도한 것으로 나타날 수 있습니다. 작은 프로젝트 또는 간단한 애플리케이션에 대해서는 오버 엔지니어링일 수 있습니다.
2. **학습곡선**: 개발자들이 이 아키텍처를 이해하고 효과적으로 사용하기 위해서는 시간이 필요할 수 있습니다.

## 객체 지향 프로그래밍 (OOP)

**객체 지향 프로그래밍(OOP, Object-Oriented Programming)** 은 상태(필드)와 행위(메서드)를 가진 객체를 중심으로 프로그램을 설계하는 프로그래밍 패러다임입니다. 객체에 역할과 책임을 부여하고, 이 객체들이 서로 협력하는 방식으로 프로그램을 구성합니다.

### 캡슐화 (Encapsulation)

객체의 상태와 행위를 하나의 단위로 묶는 것을 말합니다.

- 내부 구현은 숨기고 외부에서 접근할 수 있는 인터페이스만 제공함으로써 객체의 무결성을 보호하고 코드의 유지보수성을 높일 수 있습니다.

### 추상화 (Abstraction)

불필요한 세부 사항을 감추고 핵심적인 기능만 간추려내는 것을 말합니다.

- 객체의 공통적인 특징은 추출하여 인터페이스 또는 추상 클래스로 정의하고, 구체적인 세부 사항은 구현체에게 위임함으로써 객체의 핵심 기능에만 집중할 수 있습니다.
- **추상 클래스(Abstract Class)**: 공통된 기능을 재사용하려는 목적으로 사용됩니다. 일반 메서드와 추상 메서드를 포함할 수 있고, 인스턴스 변수를 가질 수 있습니다. is-a 관계이거나 코드의 재사용이 중요한 경우 사용합니다.
- **인터페이스(Interface)**: 구현을 강제하려는 목적으로 사용됩니다. 추상 메서드와 JDK 8부터 default 메서드를 포함할 수 있고, 상수만 가질 수 있습니다. can-do 관계이거나 다중 상속이 필요한 경우 사용합니다.

### 다형성 (Polymorphism)

하나의 인터페이스가 여러 형태로 동작할 수 있는 것을 말합니다.

- **오버로딩(Overloading)**: 클래스 내에서 같은 이름의 메서드를 여러 개 정의하는 것입니다. 매개변수의 개수, 타입, 순서가 달라야 합니다. 반환 타입만 다른 경우는 오버로딩이 성립하지 않습니다. "같은 기능을 다르게 사용하는 것"
- **오버라이딩(Overriding)**: 상위 클래스의 메서드를 하위 클래스에서 재정의하는 것입니다. 상위 클래스의 메서드 시그니처(메서드명, 매개변수 타입, 개수, 순서)와 반환 타입이 동일해야 하며, 접근 제어자는 상위 클래스와 같거나 더 넓은 범위로 변경할 수 있습니다. "상속받은 기능을 변경하는 것"

### 상속 (Inheritance)

상위 클래스의 특징을 하위 클래스가 물려받아 확장하는 것을 말합니다.

- 기존 기능을 수정하지 않고 새로운 기능을 추가할 수 있어 확장성이 뛰어나고, 중복을 제거하여 코드의 재사용성을 높일 수 있습니다.

### TDA (Tell Don't Ask) 원칙

객체의 데이터를 직접 요청하지 말고, 객체에게 필요한 동작을 수행하도록 메시지를 보내라는 원칙입니다.

- TDA 원칙을 따르면 캡슐화를 지킬 수 있고, 객체 스스로 데이터를 다루기 때문에 객체의 응집도를 높이고 객체 간 결합도를 낮출 수 있습니다.

```java
// 위반 예시 (Bad) - 객체의 데이터를 직접 가져와서 사용
Account account = new Account(BigDecimal.TEN);
BigDecimal balance = account.getBalance();
if (balance.compareTo(withdrawalAmount) < 0) {
    throw new IllegalStateException("잔액이 부족합니다.");
}
balance = balance.subtract(withdrawalAmount);
account.setBalance(balance);

// 준수 예시 (Good) - 객체에 메시지 전달
Account account = new Account(BigDecimal.TEN);
account.withdraw(withdrawalAmount); // 객체에 메시지 전달
```
