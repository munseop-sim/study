# 네트워크 인프라

프록시, 로드밸런싱, CDN, DNS, IP 주소 체계 등 네트워크 인프라와 관련된 핵심 개념을 정리한 문서입니다.

---

## 포워드 프록시 vs 리버스 프록시

### 포워드 프록시 (Forward Proxy)

클라이언트 측에 위치하여, 사용자가 인터넷에 접근할 때 중개자 역할을 하는 서버.

- **위치**: 클라이언트와 인터넷 사이
- **흐름**: 클라이언트 → 포워드 프록시 → 외부 서버
- **클라이언트 노출**: 클라이언트의 실제 IP가 숨겨지고 프록시 서버의 IP가 대신 사용됨

**주요 기능**:
- **익명성 제공**: 사용자의 실제 IP를 숨겨 개인정보 보호
- **캐싱**: 자주 요청되는 리소스를 저장해 응답 속도 향상 및 대역폭 절약
- **보안 강화**: 악성 사이트나 불법 콘텐츠에 대한 접근 차단, 바이러스/악성 코드 유입 예방

**사용 예시**: 회사 내부 네트워크에서 외부 웹사이트 접속 시 직원들의 IP를 숨기고 접근 제어

### 리버스 프록시 (Reverse Proxy)

서버 측에 위치하여 외부에서 들어오는 클라이언트의 요청을 내부 서버로 전달하는 서버.

- **위치**: 인터넷과 내부 서버 사이
- **흐름**: 클라이언트 → 리버스 프록시 → 내부 서버
- **서버 노출**: 외부에서 백엔드 서버에 직접 접근 불가, 리버스 프록시 주소만 노출

**주요 기능**:
- **로드 밸런싱**: 다수의 백엔드 서버로 트래픽 분산, 서버 과부하 방지, 고가용성 유지
- **보안**: DDoS 공격이나 해킹 시도로부터 서버 보호
- **SSL 종료**: SSL/TLS 암호화를 리버스 프록시에서 처리하여 백엔드 서버 부담 감소, 중앙화된 인증서 관리
- **캐싱 및 콘텐츠 최적화**: 정적 콘텐츠 캐싱으로 응답 속도 향상 및 서버 부하 감소

**사용 예시**: Nginx를 리버스 프록시로 사용하여 여러 WAS 서버에 요청 분산

## 타임아웃 (Timeout)

### Connection Timeout

클라이언트가 서버에 연결을 시도할 때 일정 시간 내에 연결이 이루어지지 않으면 발생하는 타임아웃.

- **발생 조건**: TCP 3-way Handshake가 일정 시간 내에 완료되지 않을 때
- **원인**: 서버 장애, 응답 지연으로 연결 자체가 맺어지지 않음

### Socket Timeout

Connection 이후 데이터 전송 중에 발생하는 타임아웃.

- **발생 조건**: 서버가 데이터를 여러 패킷으로 분할해 전송할 때, 각 패킷 사이의 시간 차이가 제한을 초과할 때
- **원인**: 서버가 일정 시간 내에 다음 패킷을 전송하지 않는 경우

### Read Timeout

클라이언트와 서버가 연결된 후 특정 I/O 작업이 일정 시간 내에 완료되지 않으면 발생하는 타임아웃.

- **발생 조건**: 서버의 응답이 지연되거나 I/O 작업이 길어져 요청이 처리되지 않을 때
- **동작**: 일정 시간 내에 데이터가 읽혀지지 않으면 클라이언트가 연결 종료

### 타임아웃이 필요한 이유

자원을 절약하기 위함. 외부 서비스 요청이 무한정 길어지면 서비스의 자원이 고갈되어 장애 발생 가능. 타임아웃 설정으로 이를 예방.

## 로드밸런싱 (Load Balancing)

### 개념

애플리케이션을 지원하는 리소스 풀에 들어오는 네트워크 트래픽을 균등하게 분산하는 것. 로드 밸런서는 애플리케이션 서버 앞단에 위치하여 클라이언트 요청을 지시하고 제어합니다.

**목적**: 애플리케이션의 가용성, 확장성, 보안 및 성능 확보

### 로드밸런싱 알고리즘

#### 라운드 로빈 (Round Robin)

모든 요청이 순서대로 처리되는 방식. 서버 A, B, C가 있으면 ABCABC 순으로 분산.

- **장점**: 구현이 쉽고 고른 분산 보장
- **단점**: 서버 부하나 응답 시간을 고려하지 않음. 서버 처리 능력이 다른 경우 비효율적

#### 가중치 라운드 로빈 (Weighted Round Robin)

각 서버에 처리 능력과 가용 자원에 따라 가중치를 할당. 가중치가 높은 서버에 더 많은 요청 전달.

- **장점**: 서버별 처리 능력 차이를 반영
- **단점**: 라운드 로빈보다 구현이 복잡. 여전히 서버 현재 상태는 고려하지 않음

#### 최소 연결 (Least Connections)

활성 연결 수가 가장 적은 서버에 요청 전달.

- **적합한 경우**: 서버 처리 능력이 비슷하지만 특정 서버에 동시 연결이 많아지는 상황
- **단점**: 서버별 처리 능력이 다른 경우 적합하지 않을 수 있음

#### 가중치 최소 연결 (Weighted Least Connections)

처리 능력이 다른 서버들이 존재할 때 가중치와 최소 연결을 결합하는 방식.

#### 최소 응답 시간 (Least Response Time)

응답 시간이 가장 빠른 서버에 요청 전달.

- **장점**: 빠른 서버에 집중하여 사용자 경험 개선
- **단점**: 서버 부하 상태, 활성 연결 수 등 다른 요소를 고려하지 않음

#### IP 해시 (IP Hash)

클라이언트 요청의 IP를 기반으로 해시값을 계산해 전달할 서버를 결정.

- **장점**: 동일 클라이언트가 항상 같은 서버로 연결 → 세션 친화성 유지에 유리
- **단점**: 상황에 따라 부하가 균등하게 분산되지 않을 수 있음

## CDN (Content Delivery Network)

### 개념

전 세계에 분산된 서버 네트워크. 사용자와 물리적으로 가까운 위치에서 정적 콘텐츠(또는 동적 콘텐츠)를 제공하여 서버 과부하를 방지하고 통신 지연을 단축합니다.

대표 서비스: CloudFront, CloudFlare

### CDN 캐싱 방식

#### Push 방식

원본 서버가 콘텐츠를 미리 CDN 서버로 전달하는 방식.

- **장점**: 적절한 시기에 정확한 콘텐츠 제공 가능
- **단점**: 지속적인 업데이트 관리 비용이 상대적으로 높음

#### Pull 방식

CDN 서버가 원본 서버로 요청을 보내 콘텐츠를 가져오는 방식. 클라이언트 요청 시 CDN에 없으면 원본 서버에서 가져와 캐시.

- **장점**: 관리 비용이 낮음
- **단점**: 초기 요청에서 원본 서버로부터 가져오는 작업으로 응답 속도 저하 가능

### CDN 도입 시 고려 사항

- **비용**: 제 3 사업자에게 비용 지불. 성능 향상이 필요한 콘텐츠만 캐싱하는 등 비용 최적화 필요
- **만료 시간**: 만료 시간이 길면 콘텐츠 신선도 저하, 짧으면 원본 서버 요청 빈번
- **CDN 장애 대응**: CDN 장애 시 원본 서버에서 직접 응답받도록 폴백(fallback) 구성
- **콘텐츠 무효화**: 오브젝트 버저닝 또는 CDN API를 통한 캐시 무효화 방식 선택

## DNS 및 정적/동적 IP

### DNS (Domain Name System)

IP 주소는 변환될 수 있고 기억하기 어렵기 때문에 도메인 주소를 사용합니다. DNS는 도메인 주소에 대응되는 원격 호스트 IP 주소를 관리하고 질의할 수 있는 시스템 또는 프로토콜입니다.

#### DNS 서버 계층 구조

- **로컬 네임 서버**: 통신사 DNS, 구글 DNS처럼 클라이언트와 가장 가까운 서버
- **루트 네임 서버**: 루트 도메인 관리. TLD 네임 서버의 주소 보유
- **TLD 네임 서버**: `.com`, `.kr`, `.net` 등 관리
- **권한 네임 서버**: 실제 기관에서 보유한 도메인 관리 (예: `maeil-mail.kr` 관리)

#### DNS 질의 과정 (예: `api.maeil-mail.kr`)

1. 클라이언트가 로컬 네임 서버에 IP 질의
2. 로컬 네임 서버 캐시에 없으면 루트 네임 서버에 질의 → `.kr` TLD 네임 서버 주소 응답
3. 로컬 네임 서버가 `.kr` TLD 네임 서버에 질의 → `maeil-mail.kr` 권한 네임 서버 주소 응답
4. 로컬 네임 서버가 권한 네임 서버에 질의 → `api.maeil-mail.kr` IP 주소 응답
5. 로컬 네임 서버가 클라이언트에게 IP 주소 전달

### 정적 IP vs 동적 IP

#### 정적 IP 할당

수동으로 IP를 설정하는 방식. IP 주소, 서브넷 마스크, 게이트웨이, DNS 주소가 필요.

- **단점**: 호스트 수가 많아질수록 번거롭고 중복 IP 입력 등 실수 유발 가능

#### 동적 IP 할당 (DHCP)

**DHCP(Dynamic Host Configuration Protocol)** 를 사용하여 IP를 자동으로 할당하는 방식. 현재 사용하지 않는 IP를 호스트에게 임대.

- IP 주소가 고정적이지 않고 변경될 수 있음

**DHCP IP 할당 과정 (4단계)**:
1. **Discover**: 호스트가 브로드캐스팅하여 DHCP 서버 탐색
2. **Offer**: DHCP 서버가 할당할 IP 주소와 임대 기간 포함한 Offer 메시지 전송
3. **Request**: 호스트가 Offer에 대한 응답으로 Request 메시지를 브로드캐스팅
4. **Acknowledgment**: DHCP 서버가 ACK 메시지로 IP 임대 승인

임대 기한 만료 전에 **DHCP Lease Renewal**을 통해 임대 기간 연장 가능.

## 클래스풀 IP 및 NAT

### 클래스풀 IP 주소 체계 (Classful Addressing)

IP 주소를 규격화된 크기별로 구분하는 방식. IPv4 주소(32비트)를 클래스별로 구분.

| 클래스 | 시작 비트 | 네트워크 주소 | 호스트 주소 | 네트워크 수 | 호스트 수 |
|--------|-----------|---------------|-------------|-------------|-----------|
| A | 0 | 1옥텟 | 3옥텟 | 2^7개 | 2^24 - 2개 |
| B | 10 | 2옥텟 | 2옥텟 | 2^14개 | 2^16 - 2개 |
| C | 110 | 3옥텟 | 1옥텟 | 2^21개 | 2^8 - 2개 |

**한계**: 클래스별 네트워크 크기가 고정되어 IP 주소 낭비 발생 (예: 컴퓨터 255대인 경우 C 클래스 사용 불가, B 클래스 사용 필요)

### 클래스리스 IP 주소 체계 (Classless Addressing)

서브넷 마스크를 이용해 네트워크 주소와 호스트 주소를 구분하는 방식. 클래스풀의 IP 낭비 문제를 해결.

- **CIDR 표기법**: `IP 주소/서브넷 마스크 비트열의 1의 수`
  - 예: `168.168.168.168/24` (서브넷 마스크 `255.255.255.0`)

### NAT (Network Address Translation)

공인 IP와 사설 IP를 변환하는 기술.

**공인 IP vs 사설 IP**:
- **공인 IP**: 고유하며 외부 네트워크에서 사용
- **사설 IP**: 사설 네트워크 내에서만 유효 (라우터가 할당)

**NAT 필요 이유**: 사설 IP만으로는 외부 네트워크와 통신 불가. NAT를 통해 사설 IP를 공인 IP로 변환.

**NAPT (Network Address Port Translation)**:

일대일 대응 NAT은 사설 IP만큼 공인 IP가 필요하므로 한계가 있음. 일반적으로 포트 기반 NAPT를 사용.

| Private IP | Private Port | Public IP | Public Port |
|------------|--------------|-----------|-------------|
| 192.168.10.2 | 8000 | 1.2.3.4 | 8001 |
| 192.168.10.3 | 8000 | 1.2.3.4 | 8002 |
