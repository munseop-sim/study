# 스레드와 프로세스

## 프로세스 vs 스레드 vs 코어 수 관계

### 스레드가 많으면?

스레드가 지나치게 많아지면 다음과 같은 문제가 발생합니다:

- **컨텍스트 스위칭 증가**: 운영체제가 스레드 간 컨텍스트 스위칭을 자주 수행해야 하여 CPU 자원이 스레드 관리에 소모됨
- **자원 경쟁**: 메모리, 캐시, 락 등의 자원을 경쟁하게 되어 성능 저하 또는 데드락 발생 가능성 증가
- **관리 복잡성**: 동기화와 상태 관리가 복잡해져 버그 발생 가능성 증가

### 프로세스가 많으면?

- **메모리 급증**: 각 프로세스는 독립된 메모리 공간을 가지므로 메모리 사용량이 급격히 증가
- **생성/관리 비용**: 프로세스 생성과 관리에 상당한 시스템 자원 소모
- **IPC 성능 저하**: 프로세스 간 통신(IPC)이 필요할 경우 성능 저하 발생
- **무거운 컨텍스트 스위칭**: 프로세스 간 컨텍스트 스위칭은 스레드보다 오버헤드가 훨씬 큼
- **생성 제한**: 운영체제는 동시에 실행할 수 있는 프로세스 수에 제한이 있음

### 코어가 많으면?

- **병렬 처리 향상**: 많은 코어를 가진 CPU는 병렬 처리 성능을 향상
- **소프트웨어 최적화 필요**: 멀티코어 환경 활용을 위해 소프트웨어가 멀티코어에 최적화되어 있어야 함
- **단일 스레드 한계**: 단일 스레드 작업이 주를 이루는 경우, 추가 코어의 이점을 제대로 활용하지 못함
- **비용/발열**: 코어 수가 많아질수록 CPU 비용, 전력 소비, 발열 관리가 복잡해짐

> **결론**: 스레드, 프로세스, 코어의 수가 많다고 무조건 성능이 향상되지 않으며, 작업 특성에 맞는 적절한 수를 설정하는 것이 중요합니다.

## 컨텍스트 스위칭(Context Switching)

### 정의

**컨텍스트 스위칭**은 CPU나 코어에서 실행 중이던 프로세스나 스레드가 다른 프로세스나 스레드로 교체되는 과정입니다. 멀티태스킹 시스템에서 여러 작업을 효율적으로 관리하기 위한 필수 메커니즘입니다.

### 컨텍스트(Context)란?

프로세스나 스레드의 현재 상태를 의미합니다:
- CPU의 레지스터 상태 (프로그램 카운터, 스택 포인터 등)
- 메모리 상태
- 실행 중단 후 재시작 시 필요한 모든 정보

### 컨텍스트 스위칭이 발생하는 시점

1. **Time Slice(Quantum) 소진**: 각 프로세스/스레드에 할당된 CPU 시간이 소진됐을 때
2. **I/O 작업 필요**: 프로세스/스레드가 I/O 작업을 수행해야 할 때
3. **우선순위 변경**: 더 높은 우선순위의 작업이 발생했을 때

### 컨텍스트 스위칭 수행 주체

**운영체제의 커널(Kernel)**이 수행합니다.

### 프로세스 컨텍스트 스위칭 과정

1. 현재 프로세스의 레지스터 상태와 메모리 정보를 저장
2. 스케줄러가 선택한 다음 프로세스의 저장된 상태를 로드
3. CPU는 새로운 프로세스의 실행을 시작
4. **추가 메모리 처리**: 새로운 프로세스의 가상 메모리 주소 체계 설정, MMU(Memory Management Unit)와 TLB(Translation Lookaside Buffer) 업데이트

### 스레드 컨텍스트 스위칭 과정

1. 현재 스레드의 레지스터 상태를 저장
2. 동일한 프로세스 내에서 실행할 다음 스레드의 상태를 로드
3. CPU는 새로운 스레드의 실행을 시작
4. **메모리 처리 생략**: 동일한 프로세스 내 스레드 간 전환이므로 메모리 관련 추가 처리 불필요

### 프로세스 vs 스레드 컨텍스트 스위칭 비교

| 구분 | 프로세스 컨텍스트 스위칭 | 스레드 컨텍스트 스위칭 |
|------|--------------------------|------------------------|
| 공통점 | 커널 모드에서 실행, CPU 레지스터 상태 교체 | 커널 모드에서 실행, CPU 레지스터 상태 교체 |
| 메모리 처리 | MMU 업데이트, TLB 캐시 초기화 필요 | 동일 프로세스 공유 메모리이므로 불필요 |
| 비용 | 높음 | 낮음 |

### 스레드 컨텍스트 스위칭이 더 빠른 이유

같은 프로세스에 속한 스레드들은 메모리 영역을 공유하기 때문에:
- MMU의 새로운 주소 체계 설정 불필요
- TLB(가상-실제 메모리 주소 매핑 캐시) 비우기 불필요
- **CPU의 상태 정보만 교체**하면 됨

> TLB 캐시를 비우지 않으면 이전 프로세스의 주소에 접근할 가능성이 있으므로, 프로세스 스위칭 시에는 반드시 수행해야 합니다.

## 멀티태스킹(Multi-Tasking)

### 단일 프로세스 시스템의 한계

한 번에 하나의 프로그램만 실행하며, 다른 프로그램 실행 시 현재 프로그램을 먼저 종료해야 합니다.

**단점**: CPU 사용률이 좋지 않음. 프로세스가 I/O 작업을 하게 되면 CPU가 아무런 일을 하지 않는 상태로 대기합니다.

### 멀티 프로그래밍(Multi-Programming)

단일 프로세스의 CPU 낭비 문제를 개선합니다.

- 하나의 프로세스가 I/O 작업으로 대기하게 되면 다른 프로세스가 실행되어 CPU가 아무 일도 하지 않는 시간을 줄임
- **주된 목적**: CPU 사용률 극대화

**단점**: 하나의 프로세스가 CPU를 사용하는 작업을 계속 점유하게 되면 다른 프로세스가 대기해야 하는 문제 발생

### 멀티태스킹(Multi-Tasking)

두 개 이상의 프로세스가 CPU를 사용할 때 **아주 짧은 시간(Quantum) 동안만 번갈아가며 실행**될 수 있도록 합니다.

- **목적**: 프로세스의 응답 시간 최소화
- 아주 짧은 타임 슬라이스 안에서 여러 프로세스들이 교대로 실행
- 사용자 입장에서 **마치 여러 프로그램이 동시에 실행되는 것처럼** 느껴짐

### 멀티 프로그래밍 vs 멀티태스킹 비교

| 구분 | 멀티 프로그래밍 | 멀티 태스킹 |
|------|----------------|-------------|
| 공통점 | 동시에 여러 프로그램 실행 | 동시에 여러 프로그램 실행 |
| 차이점 | CPU가 유휴 상태가 되면 다른 프로세스 실행 | CPU 타임을 짧게 쪼개서 프로세스들이 번갈아 실행 |
| 목적 | CPU 사용률 극대화 | 응답 시간 최소화 |

## 멀티스레딩(Multi-Threading)

### 멀티태스킹 시스템의 한계

1. **하나의 프로세스가 동시에 여러 작업을 수행하지 못함**
2. **무거운 프로세스 간 컨텍스트 스위칭**
3. **프로세스 간 데이터 공유의 어려움** (각 프로세스는 독립적인 메모리 공간 사용)

### 스레드(Thread)의 등장

- 한 프로세스 내에서 여러 작업을 동시에 실행할 수 있도록 도와줌
- 하나의 프로세스는 하나 이상의 스레드를 가질 수 있음
- 현재는 **스레드가 CPU에서 실행되는 최소 단위**
- 각 스레드는 고유한 Stack, 포인터, 프로그램 카운터를 가짐
- 같은 프로세스 내 스레드들은 **코드, 힙 메모리 등을 공유**

### 멀티스레딩의 주요 특징

#### 1. 경량화된 실행 단위
- 스레드 간 컨텍스트 스위칭은 프로세스 간 전환보다 훨씬 가볍고 빠름
- 공유 메모리가 있어 재설정해야 할 데이터의 양이 적음

#### 2. 효율적인 데이터 공유
- 같은 프로세스 내 스레드들은 힙 영역 등 주요 메모리 공간을 공유
- IPC와 같은 복잡한 메커니즘 없이 데이터 전달 가능
- 단, **동시 접근 시 동기화 문제** 여전히 존재

#### 3. 응답성 및 처리 성능 향상
- I/O 작업과 CPU 집약적 작업을 분리하여 동시에 처리
- 멀티코어 환경에서 각 스레드를 개별 코어에 할당하여 병렬 처리 가능

### 멀티스레딩의 단점

- **경쟁 상태(Race Condition)**, **교착 상태(Deadlock)** 등의 동기화 문제 발생 가능
- 동기화 처리 과정이 복잡하고 디버깅이 어려움
- 스레드 관리를 소홀히 하면 **시스템 자원이 과도하게 사용될 위험**

### 멀티스레딩 vs 멀티프로세싱

| 구분 | 멀티스레딩 | 멀티프로세싱 |
|------|-----------|-------------|
| 정의 | 하나의 프로세스 내 여러 스레드로 여러 작업 동시 실행 | 두 개 이상의 프로세서/코어로 여러 프로세스를 동시 실행 |
| 메모리 공유 | 프로세스 내 메모리 공유 | 독립적인 메모리 공간 |
| 컨텍스트 스위칭 비용 | 낮음 | 높음 |
| 데이터 공유 | 쉬움 (공유 메모리) | 어려움 (IPC 필요) |

## ThreadLocal

### 정의

`ThreadLocal`은 Java에서 **각 스레드마다 독립적인 변수를 저장**할 수 있도록 도와주는 클래스입니다.

각 스레드는 자신만의 `ThreadLocalMap`을 가지고 있으며, `ThreadLocal`을 키로 사용하여 값을 저장합니다.

### Spring 생태계에서의 활용

- **`TransactionSynchronizationManager`**: 트랜잭션 동기화 관리
- **`SecurityContextHolder`**: 사용자 인증 정보 관리
- **`RequestContextHolder`**: 웹 요청의 attribute 관리

### 장점

- 각 스레드는 ThreadLocal에 접근할 때 다른 스레드와 격리된 값을 가짐
- `synchronized` 등 동기화 없이 thread-safe 보장

### 주의사항

1. **스레드풀 재사용 문제**: 스레드가 재사용될 때 이전 값이 남아있으면 잘못된 데이터 참조 가능
   - 스레드가 끝나는 시점에 **`remove()` 메서드를 호출**하여 값 제거 필수

2. **비동기 작업 시 주의**: `@Async` 사용 시 새로운 스레드에서 실행되므로 기존 ThreadLocal 값을 참조할 수 없음
   - Spring 4.3 이상의 `TaskDecorator`를 사용하여 값 복사 가능

### 대안

- 메서드 인자로 값 전달
- `ConcurrentHashMap`과 같은 thread-safe 자료구조 사용
- Spring `@RequestScope` 어노테이션 (HTTP 요청별 데이터 관리)

### NamedThreadLocal

Spring에서 제공하는 `ThreadLocal`의 확장 클래스. 디버깅을 쉽게 하기 위해 이름을 부여할 수 있도록 설계되었습니다.
