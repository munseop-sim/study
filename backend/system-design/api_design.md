# API 설계 (API Design)

## REST (Representational State Transfer)

**REST(Representational State Transfer)** 는 자원의 표현을 이용하여 상태를 주고받는 것을 의미합니다.

- **자원(Resource)**: 소프트웨어가 관리하는 모든 것
- **자원의 표현**: 자원을 나타내기 위한 이름. 예를 들어, 서버가 관리하는 주문 데이터는 `order`라고 표현할 수 있습니다.
- 최근에는 일반적으로 자원의 상태를 나타내기 위해 **JSON 포맷**을 사용합니다.

REST는 네트워크 상에서 클라이언트와 서버의 통신 방식 중 하나이며, **HTTP 프로토콜**을 사용합니다. 구체적으로는 HTTP URI를 활용하여 자원을 명시하고 HTTP METHOD를 통해 CRUD 연산을 적용하는 것을 의미합니다.

**API(Application Programming Interface)** 란 컴퓨터 프로그램 간 정보를 주고받을 수 있도록 하는 일종의 출입구와 같은 역할을 수행합니다. API가 REST 기반으로 구현되어 있다면, 이를 **REST API**라고 부릅니다.

### REST의 장점

- 서버와 클라이언트의 역할을 명확하게 분리해 줍니다.
- HTTP 프로토콜을 따르는 모든 플랫폼에서 사용할 수 있습니다.
- CURL, Postman을 사용하여 간단하게 테스트할 수 있습니다.
- JSON과 같은 텍스트 포맷은 자기 서술적이며, 메시지 소비자가 자신이 관심이 있는 값만 골라서 사용하고 나머지는 무시하면 되므로 메시지 구조가 자주 바뀌어도 하위 호환성을 보장하는 것이 유리합니다.

### REST의 단점

- 요청 및 응답 스타일의 통신만 지원합니다.
- HTTP 메서드로 행위를 표현하기 어려운 경우도 존재합니다.
- 요청 한 번으로 여러 자원을 가져오기 어렵다는 단점이 존재합니다.
- JSON과 같은 텍스트 포맷은 메시지가 다소 길다는 것이 단점입니다. 메시지가 길다면 네트워크 트래픽을 더욱 사용하며 전송 속도가 느릴 수 있으며, 메시지를 해석하는 데에 오버헤드가 발생할 수 있습니다.

## 시스템 간 비동기 연동 방식

분리된 시스템 간의 비동기 연동은:
- 시스템 간의 결합도를 낮출 수 있습니다.
- 호출된 시스템의 응답을 기다리지 않으므로 더욱 빨리 사용자의 요청에 응답할 수 있다는 장점이 있습니다.

### 1. 메시징 시스템 활용 방식

두 시스템 사이에 메시징 시스템을 두어 비동기로 연동하는 방식입니다.

**흐름:**
1. 한 시스템에서 메시지를 생성해서 메시징 시스템에 송신
2. 다른 시스템에서 메시징 시스템으로부터 메시지를 읽어와 처리

대표적인 메시징 시스템: **Kafka, RabbitMQ**

**장점:**
- 처리량이 높습니다.

**단점/고려사항:**
- 메시지 유실에 대한 고민이 필요합니다.
- 메시지 소비 순서 보장 문제가 있습니다.
- 트랜잭션 처리 문제: 1개의 트랜잭션 내에 메시지 전송과 데이터베이스 삽입이 존재할 때, 데이터베이스 삽입이 실패했는데 메시지는 전송되거나, 그 반대의 경우를 고려해야 합니다. 두 작업을 원자적으로 수행할 방법이 필요합니다.

### 2. 데이터베이스 활용 방식

데이터베이스를 메시징 시스템처럼 사용하는 방법입니다.

**흐름:**
1. 한 시스템에서 데이터베이스 테이블에 필요한 메시지 레코드를 추가
2. 연동 시스템이 테이블을 주기적으로 읽음
3. 새로운 메시지가 추가되면 연동 시스템은 다른 시스템으로 메시지를 전송

**장점:**
- 트랜잭션과 메시지 순서가 보장됩니다.
- 메시지 유실에 대한 걱정이 없습니다.

**단점:**
- 범용성이 떨어질 수 있습니다. 메시지 형식이 빈번히 변경될 때마다 메시지 레코드에 대한 스키마도 변경해야 합니다.
- 삭제 정책과 읽기 빈도 등 추가적으로 고민해야 할 부분들이 존재합니다.

### 3. CDC (Change Data Capture) 활용 방식

데이터베이스의 변경 사항을 조회하여, 이를 다른 시스템에 전파하는 방식입니다.

예를 들어, 별도의 시스템이 변경 감지 대상 데이터베이스의 **바이너리 로그(Binary Log)** 를 조회하여 변경을 전파하도록 구현할 수 있습니다.

**장점:**
- 트랜잭션이 보장됩니다.
- 메시지를 생성하거나 별도로 저장할 필요가 없으니 상대적으로 애플리케이션 로직이 단순합니다.

**단점:**
- 변경 로그만 존재할 뿐 왜 바뀌었는지에 대한 추가적인 정보가 없기 때문에 사용하는 데 제약이 존재할 수 있습니다.
