# 관측 가능성 (Observability)

---

## 로그 vs 메트릭

> 출처: https://www.maeil-mail.kr/question/305

### 개념 비교

시스템의 상태를 파악하기 위한 두 가지 주요 데이터 유형입니다.

| 구분 | 로그 (Log) | 메트릭 (Metric) |
|---|---|---|
| **형태** | 이벤트 기록 (텍스트) | 수치 데이터 (숫자) |
| **목적** | 무슨 일이 일어났는지 상세 추적 | 시스템 상태를 수치로 요약 |
| **저장 방식** | 이벤트 발생 시마다 기록 (append) | 시계열 데이터로 집계 |
| **조회 방식** | 특정 이벤트 검색/필터링 | 시간 범위 집계, 그래프 시각화 |
| **용량** | 대용량 (이벤트마다 기록) | 소용량 (집계된 수치) |
| **대표 도구** | ELK Stack, Loki | Prometheus, Micrometer |

### 로그 (Log)

로그는 시스템에서 발생한 **이벤트의 시간 순서대로 기록**한 데이터입니다.

**특징:**
- 각 이벤트의 상세 정보를 포함 (타임스탬프, 레벨, 메시지, 컨텍스트)
- 특정 요청이나 오류를 추적하는 데 유용
- 비정형 텍스트 데이터로 저장 용량이 큼

**로그 레벨:**

| 레벨 | 용도 |
|---|---|
| ERROR | 즉각적인 조치가 필요한 오류 |
| WARN | 잠재적인 문제, 주의 필요 |
| INFO | 정상적인 운영 정보 |
| DEBUG | 디버깅용 상세 정보 |
| TRACE | 가장 상세한 진단 정보 |

```java
// Spring Boot 로그 예시
log.info("주문 생성 완료 - orderId: {}, userId: {}", orderId, userId);
log.error("결제 처리 실패 - orderId: {}", orderId, exception);
```

**구조화된 로그 (Structured Logging):**
```json
{
  "timestamp": "2024-01-15T10:30:00Z",
  "level": "ERROR",
  "message": "결제 처리 실패",
  "orderId": "12345",
  "userId": "user_001",
  "exception": "PaymentException"
}
```

### 메트릭 (Metric)

메트릭은 시스템의 상태를 **수치로 집계한 시계열 데이터**입니다.

**특징:**
- 숫자 형태의 측정값 (CPU 사용률, 요청 수, 응답 시간 등)
- 시간에 따른 변화 추이 파악 및 알림(Alert) 설정에 유용
- 로그에 비해 저장 용량이 적고 집계/조회가 빠름

**메트릭 타입:**
- **Counter**: 단조 증가 (HTTP 요청 수, 오류 수)
- **Gauge**: 현재 상태 (활성 커넥션 수, 큐 크기)
- **Histogram**: 분포 측정 (응답 시간 분포, 요청 크기 분포)
- **Summary**: 백분위수 계산 (p50, p95, p99 응답 시간)

### 언제 로그를, 언제 메트릭을 사용하나?

- **로그**: "왜 이 오류가 발생했는가?" → 특정 요청의 상세 원인 분석
- **메트릭**: "시스템이 얼마나 건강한가?" → 전반적인 성능 및 상태 모니터링

### 관측 가능성의 3요소 (Three Pillars of Observability)

1. **로그(Logs)**: 이벤트 상세 기록
2. **메트릭(Metrics)**: 수치 기반 상태 측정
3. **트레이스(Traces)**: 분산 시스템에서 요청의 전체 흐름 추적

## RAID (Redundant Array of Independent Disks)

> 출처: https://www.maeil-mail.kr/question/304

### 개념

**RAID(Redundant Array of Independent Disks)** 는 여러 개의 디스크를 묶어 **하나의 논리적인 디스크처럼 사용**하는 기술입니다. 데이터 안전성(내결함성), 성능 향상, 또는 두 가지 모두를 목적으로 사용합니다.

### RAID 레벨별 특징

#### RAID 0 (스트라이핑, Striping)

데이터를 여러 디스크에 **분산 저장**하여 성능을 향상시킵니다.

```
데이터: [A1][A2][A3][A4]
디스크1: [A1][A3]
디스크2: [A2][A4]
```

- **장점**: 읽기/쓰기 성능 최대화 (디스크 수에 비례)
- **단점**: 디스크 하나라도 고장 시 **전체 데이터 손실** (내결함성 없음)
- **용도**: 영상 편집 등 고성능이 필요하고 데이터 손실을 감수할 수 있는 경우

#### RAID 1 (미러링, Mirroring)

동일한 데이터를 **두 개 이상의 디스크에 복제**합니다.

```
디스크1: [A1][A2][A3]  (원본)
디스크2: [A1][A2][A3]  (미러)
```

- **장점**: 디스크 하나 고장 시에도 **데이터 보호** (높은 내결함성)
- **단점**: 스토리지 효율 50% (전체 용량의 절반만 사용 가능), 쓰기 성능 향상 없음
- **용도**: 중요 데이터 보호가 최우선인 경우

#### RAID 5 (스트라이핑 + 패리티)

데이터와 **패리티(오류 복구 정보)** 를 여러 디스크에 분산 저장합니다.

```
디스크1: [A1][B1][P_C]
디스크2: [A2][P_B][C1]
디스크3: [P_A][B2][C2]
(P = 패리티)
```

- **장점**: 디스크 1개 고장 허용, 읽기 성능 향상, RAID 1보다 스토리지 효율 높음
- **단점**: 쓰기 시 패리티 계산으로 쓰기 성능 저하, 2개 이상 디스크 고장 시 데이터 손실
- **최소 디스크 수**: 3개
- **용도**: 성능과 안전성의 균형이 필요한 경우 (가장 널리 사용)

#### RAID 6 (스트라이핑 + 이중 패리티)

RAID 5와 유사하나 **패리티를 2개** 저장합니다.

- **장점**: 디스크 **2개 동시 고장**까지 허용
- **단점**: RAID 5보다 쓰기 성능 더 낮음, 최소 4개 디스크 필요
- **용도**: 대용량 스토리지에서 높은 내결함성이 필요한 경우

#### RAID 10 (RAID 1+0, 미러링 + 스트라이핑)

RAID 1(미러링)과 RAID 0(스트라이핑)을 **결합**한 방식입니다.

```
[미러 쌍 1]      [미러 쌍 2]
디스크1 디스크2  디스크3 디스크4
  [A1]   [A1]     [A2]   [A2]
  [A3]   [A3]     [A4]   [A4]
← 스트라이핑으로 연결 →
```

- **장점**: **높은 성능 + 높은 내결함성** 동시 달성, 같은 미러 쌍에서 동시 고장만 아니면 복구 가능
- **단점**: 최소 4개 디스크 필요, 스토리지 효율 50%
- **용도**: 성능과 안전성 모두 중요한 데이터베이스 서버

### RAID 레벨 비교표

| RAID | 최소 디스크 | 내결함성 | 읽기 성능 | 쓰기 성능 | 스토리지 효율 |
|---|---|---|---|---|---|
| RAID 0 | 2 | 없음 | 매우 높음 | 매우 높음 | 100% |
| RAID 1 | 2 | 1개 고장 | 높음 | 동일 | 50% |
| RAID 5 | 3 | 1개 고장 | 높음 | 낮음 | (N-1)/N |
| RAID 6 | 4 | 2개 고장 | 높음 | 매우 낮음 | (N-2)/N |
| RAID 10 | 4 | 각 미러 쌍 1개 | 매우 높음 | 높음 | 50% |

### 소프트웨어 RAID vs 하드웨어 RAID

| 구분 | 소프트웨어 RAID | 하드웨어 RAID |
|---|---|---|
| 처리 주체 | OS/CPU | 전용 RAID 컨트롤러 |
| 비용 | 낮음 | 높음 |
| 성능 | CPU 자원 소비 | CPU 부하 없음 |
| 예시 | Linux mdadm, ZFS | LSI, Adaptec 컨트롤러 |
