# 인프라 & 운영 (Infrastructure & Operations)

## CI/CD 파이프라인

### CI (Continuous Integration) - 지속적 통합

개발자가 작성한 작은 코드 변경을 코드 베이스에 통합합니다. 변경한 부분이 통합되면, 자동으로 새로운 시스템을 빌드하고 현재 시스템에 존재하는 모든 테스트를 실행합니다. 만약 이전에 동작했던 어떤 부분이 망가졌다면, 개발자는 해당 부분을 다시 수정합니다.

**핵심 목표:**
- 소프트웨어의 품질 개선
- 새로운 소프트웨어의 변경 사항을 검증하는 데 소요되는 시간 단축
- 버그 조기 발견

### CD (Continuous Delivery / Continuous Deployment)

- **지속적 배포(Continuous Deployment)**: 지속적 통합을 통해서 빌드된 코드(빌드 아티팩트)를 프로덕션 환경에 **자동으로** 배포하는 것을 의미합니다.
- **지속적 전달(Continuous Delivery)**: 빌드 아티팩트를 프로덕션 환경에 바로 배포하기 위해서 **수동으로 작업**해야 한다는 점에서 지속적 배포와 차이가 있습니다.

CD 과정에는 빌드 아티팩트를 관리 및 저장하는 공간이 필요할 수도 있습니다. 예를 들어, AWS S3, Docker Registry, Nexus를 사용할 수 있습니다.

**CI/CD 파이프라인 구축 도구:** Jenkins, Travis CI, Github Actions 등

## 스케일 아웃 vs 스케일 업

기존 개발하고 있던 서비스의 서버가 한계에 도달하는 경우, 스케일 업(Scale-Up) 혹은 스케일 아웃(Scale-Out)을 고려할 수 있습니다.

### 스케일 업 (Scale-Up) - 수직 확장

기존의 서버를 더욱 높은 사양으로 업그레이드하는 것을 의미합니다.

예시: AWS EC2 t2.micro → t2.small 업그레이드

**장점:**
- 상대적으로 간단하게 서버의 성능을 향상시킬 수 있습니다.

**단점:**
- 특정 서버를 무한정 업그레이드할 수 없습니다.
- 장애에 대한 자동복구(failover)나 다중화(redundancy) 방안을 제시하지 않습니다.
- 향후 사용량을 미리 추정하여 고사양 서버를 확보하는 경우, 실제 필요한 사양보다 과한 사양의 장비를 확보할 수 있기 때문에 비용적인 손실이 존재할 수 있습니다.

### 스케일 아웃 (Scale-Out) - 수평 확장

비슷한 사양의 장비를 추가하여 수평으로 확장하는 방식입니다. 서버로 들어오는 많은 요청을 비슷한 사양의 서버 n대로 분산시켜 성능을 향상시킵니다.

**장점:**
- 그때그때 필요한 만큼 서버를 추가할 수 있으므로 상대적으로 스케일 업 방식보다 비용 효율적일 수 있습니다.
- 특정 서버의 장애 발생 상황에서도 스케일 업 방식보다 가용성이 높습니다.

**단점:**
- n대의 서버를 관리해야 하므로 관리 포인트가 늘어납니다.
- 각 서버에 부하를 분산하기 위한 로드 밸런싱에 대한 고민이 추가로 필요합니다.

## SPOF (Single Point of Failure) - 단일 장애 지점

**단일 장애 지점(Single Point of Failure, SPOF)** 이란 전체 시스템에서 제대로 동작하지 않는 경우, 전체 시스템이 중단되는 특정 구성 요소를 의미합니다.

### 가용성 (Availability)

서버와 네트워크, 프로그램 등 정보 시스템이 정상적으로 사용할 수 있는 정도를 **가용성(Availability)** 이라고 합니다.

```
가용성 = 정상적인 사용 시간(Uptime) / 전체 사용 시간(Uptime + Downtime)
```

가용성이 99.999% 처럼 높은 경우에 **고가용성(High Availability, HA)** 이라 합니다. 시스템이 고가용성을 만족하기 위해서는 SPOF를 식별하고, 개선하는 작업이 필요합니다.

### SPOF 예시 및 개선 방법

**예시 상황:**
1. API Server 1대를 운용합니다.
2. DB는 Master Replica 구성을 사용합니다.
3. Master DB는 1대이며, 3대의 Replica DB를 사용하고 있습니다. (Failover는 지원됩니다.)

**SPOF 식별:** 현재 서버는 단일 서버로 구성되어 있습니다. OOM(Out Of Memory), 네트워크 장애, 자원 고갈, 하드웨어 장애와 같은 문제가 발생할 경우 서버의 고장이 전체 시스템의 고장으로 이어질 수 있습니다.

**개선 방법:**
- **서버 이중화**: 동일한 애플리케이션을 여러 서버에 배포하여 한 서버가 다운되더라도 다른 서버가 서비스를 제공할 수 있도록 합니다.
- **로드 밸런서 사용**: 서버에 들어오는 트래픽을 여러 서버로 분산시켜주는 역할을 합니다. 특정 서버가 고장나는 경우에는 로드 밸런서에서 해당 서버로 향하는 요청을 다른 정상 서버로 전달합니다.

### 서버 이중화 시 점검 사항

- **동시성 문제**: 자바의 `synchronized`, `ReentrantLock`은 단일 서버에서만 유효합니다. 이중화 환경에서는 분산 잠금이나 DB 잠금을 사용하는 것이 적절합니다.
- **세션 불일치 문제**: 세션 기반 인증을 사용하고 있다면, 세션 클러스터링이나 스티키 세션, 토큰 기반 인증, 외부 세션 저장소 등을 고려해야 합니다.
- **로그 및 메트릭 수집**: 여러 서버에서 생성되는 데이터를 수집 및 통합하여 관리할 필요성이 생깁니다.
- **로드 밸런싱 알고리즘**: 비효율적인 알고리즘은 특정 서버에만 요청이 몰릴 수 있습니다.
- **배포 전략**: 서버가 늘어날수록 동일한 서비스의 다양한 버전이 운영될 수 있으며, 배포 시간 증가나 배포 중 장애 대응이 복잡해질 수 있습니다.

## 헬스체크 (Health Check)

**헬스체크(Health Check)** 는 현재 서버의 상태가 정상인지 파악하는 것을 의미합니다.

**구현 방법:**
- API 엔드포인트를 호출하는 방식
- 특정 포트로 TCP 연결을 시도하는 방식
- 스프링 액추에이터(Spring Actuator)를 활용하는 방식

### 헬스체크의 필요성

- 최신 코드를 배포할 때 신규 배포가 정상적으로 이뤄졌는지 확인할 수 있습니다.
- 장애를 감지하여 대응할 수 있습니다.

**장애 대응 예시**: 로드 밸런서가 존재하고 트래픽 분산 대상 서버 2대(A, B)가 있을 때, A 서버의 헬스체크 결과가 비정상으로 판단되면 로드 밸런서는 A 서버를 트래픽 분산 대상에서 제외하고, 이후 B 서버로만 요청을 전달합니다.

## 외부 서비스 장애 대응 패턴

동기 방식으로 외부 서비스를 호출할 때 외부 서비스 장애로 인해 응답이 오래 걸린다면, 외부 API 응답으로 대기하는 자원들이 운영 서버 내부에 쌓이면서 성능에 악영향을 줄 수 있습니다.

### 1. 타임아웃 (Timeout) 설정

가장 기본적인 방법으로, 타임아웃 설정을 통해 응답 대기 시간을 제한합니다.

- **커넥션 타임아웃 (Connection Timeout)**: 서버에 연결하는 데 걸리는 최대 시간
- **리드 타임아웃 (Read Timeout)**: 연결 후 데이터를 읽는 데 걸리는 최대 시간
- **HTTP 커넥션 풀 타임아웃**: 커넥션 풀에서 커넥션을 획득하는 데 걸리는 최대 시간

참고: [Connection Timeout과 Read Timeout 살펴보기](https://alden-kang.tistory.com/20)

### 2. 벌크헤드 패턴 (Bulkhead Pattern)

```
문제 상황:
1. A 서비스, B 서비스, C 서비스 연동 코드가 HTTP 커넥션 풀을 공유한다.
2. A 서비스의 장애로 응답 시간 지연이 발생하는 경우
   2-1. 풀에 남은 커넥션이 점점 줄어든다.
   2-2. 풀에서 커넥션을 구하는 대기 시간이 증가한다.
   2-3. B, C 서비스에 대한 연동도 함께 대기한다.
```

**벌크헤드 패턴**은 기능의 종류마다 자원 사용을 분리하는 것을 의미합니다. 자원을 격리하여 서비스 일부에 장애가 발생해도 전체로 전파되지 않도록 보장해 주는 패턴입니다.

위 예시에서는 외부 서비스마다 다른 HTTP 커넥션 풀을 사용하도록 벌크헤드 패턴을 적용할 수 있습니다. 서로 다른 커넥션 풀을 사용하기 때문에 A 서비스에 문제가 발생해도 B, C의 영향을 최소화할 수 있습니다.

참고: [Bulkhead 패턴을 사용한 장애 격리](https://hudi.blog/bulkhead-pattern/)

### 3. 서킷 브레이커 (Circuit Breaker)

지속되는 외부 서비스 장애로 타임아웃에 의한 서비스 에러가 발생할 수 있습니다. 외부 서비스에 장애가 발생했음에도 운영 서버는 계속 요청을 보내게 되어, 불필요하게 응답 시간이 저해되고, 처리량도 감소하게 됩니다.

**서킷 브레이커**는 오류가 지속되는 경우 일정 시간 동안 기능 실행을 차단할 수 있습니다. 빠른 실패(Fail Fast)를 도와주기 때문에 외부 서비스 장애에 의한 응답 시간 증가를 예방할 수 있습니다.

```
Closed (정상) → 요청 허용, 오류율 모니터링
Open   (차단) → 오류율 임계값 초과 시, 일정 시간 동안 요청 차단 (빠른 실패)
Half-Open     → 일정 시간 후 일부 요청 허용하여 복구 여부 확인
```
