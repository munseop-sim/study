## 1. 왜 용량 산정이 필요한가

시스템 설계에서 용량 산정은 기술 선택의 근거다.
"Redis를 쓴다", "Kafka를 둔다", "DB를 샤딩한다" 같은 결정은 트래픽, 저장량, 지연 시간 목표가 있어야 판단할 수 있다.

용량 산정은 정확한 예측이 아니라 **합리적인 가정과 병목 후보를 드러내는 작업**이다.

---

## 2. 기본 입력값

최소한 다음 값을 가정한다.

| 항목 | 예시 |
|---|---|
| DAU | 1,000,000명 |
| 1인당 일 요청 수 | 20회 |
| 읽기:쓰기 비율 | 9:1 |
| 피크 계수 | 평균의 5배 |
| 평균 응답 크기 | 2KB |
| 이벤트 크기 | 1KB |
| 보존 기간 | 원본 데이터 3년, 로그 30일 |

피크 계수는 서비스 성격에 따라 다르다.
출퇴근, 점심시간, 이벤트 오픈, 월급날처럼 트래픽이 몰리는 도메인은 더 크게 잡는다.

---

## 3. QPS 계산

```
일 요청 수 = DAU * 1인당 일 요청 수
평균 QPS = 일 요청 수 / 86,400
피크 QPS = 평균 QPS * 피크 계수
```

예:

```
DAU = 1,000,000
1인당 일 요청 수 = 20
일 요청 수 = 20,000,000
평균 QPS = 20,000,000 / 86,400 = 약 231
피크 QPS = 231 * 5 = 약 1,155
```

읽기:쓰기 비율이 9:1이면:

```
읽기 피크 QPS = 1,040
쓰기 피크 QPS = 115
```

---

## 4. 저장소 용량 계산

```
일 저장량 = 일 생성 건수 * 레코드 크기
월 저장량 = 일 저장량 * 30
보존 용량 = 월 저장량 * 보존 개월 수 * 복제 계수
```

예:

```
주문 이벤트 1건 = 1KB
일 이벤트 수 = 5,000,000
일 저장량 = 약 5GB
3년 보존 = 5GB * 365 * 3 = 약 5.5TB
복제 계수 3 = 약 16.5TB
```

주의:
- 인덱스 크기는 원본 데이터와 별도로 증가한다.
- JSON payload는 예상보다 커지기 쉽다.
- 압축률은 데이터 형태에 따라 다르므로 보수적으로 잡는다.
- DB 백업, binlog, Kafka replication까지 포함해야 한다.

## 5. 네트워크 대역폭 계산

```
초당 전송량 = QPS * 평균 응답 크기
```

예:

```
피크 QPS = 1,155
평균 응답 크기 = 2KB
초당 응답 트래픽 = 2.3MB/s
```

Kafka 이벤트:

```
피크 이벤트 TPS = 10,000
이벤트 크기 = 1KB
replication.factor = 3
브로커 쓰기 트래픽 = 10,000 * 1KB * 3 = 약 30MB/s
```

네트워크 계산은 평균뿐 아니라 burst를 고려해야 한다.

---

## 6. DB 커넥션 풀 산정

커넥션 풀 크기는 "크면 좋은 값"이 아니다.
DB가 동시에 처리할 수 있는 쿼리 수보다 커지면 context switching과 lock contention이 증가한다.

단순 산정:

```
필요 동시 처리 수 = 피크 QPS * 평균 DB 점유 시간
```

예:

```
피크 QPS = 1,000
평균 DB 점유 시간 = 20ms = 0.02s
필요 커넥션 = 1,000 * 0.02 = 20
```

여기에 약간의 버퍼를 더해 25~30 정도로 시작하고, 실제 metric으로 조정한다.

봐야 할 지표:
- HikariCP active/idle/pending
- DB CPU
- slow query
- lock wait
- connection timeout

## 7. 캐시 용량 산정

캐시는 hit ratio 목표와 hot key 크기를 기준으로 잡는다.

```
필요 메모리 = 캐시 엔트리 수 * 평균 엔트리 크기 * 오버헤드 계수
```

예:

```
상품 1개 캐시 = 3KB
핫 상품 수 = 1,000,000
원본 크기 = 3GB
Redis 오버헤드 계수 = 1.5~2
필요 메모리 = 4.5~6GB
```

주의:
- Redis는 maxmemory와 eviction policy를 반드시 설정한다.
- TTL이 동시에 만료되면 cache stampede가 발생할 수 있다.
- 큰 value는 네트워크와 serialization 비용을 증가시킨다.

---

## 8. 병목 후보 찾기

용량 산정 후 다음 질문에 답해야 한다.

- 읽기 병목인가, 쓰기 병목인가?
- DB가 먼저 한계에 도달하는가, 외부 API가 먼저 한계에 도달하는가?
- 단일 hot key나 hot partition이 있는가?
- 큐에 적체가 생겼을 때 얼마나 버틸 수 있는가?
- 장애 시 retry가 원래 트래픽을 몇 배로 증폭시키는가?

Retry 증폭 예:

```
원래 QPS = 1,000
재시도 2회
외부 API 장애
최대 요청량 = 3,000 QPS
```

재시도 정책은 용량 산정에 포함해야 한다.

---

## 9. 실무 체크리스트

- 모든 숫자에는 가정과 출처를 함께 적는다.
- 평균 QPS와 피크 QPS를 분리한다.
- 읽기/쓰기 비율을 분리한다.
- 저장량은 원본, 인덱스, 복제, 백업을 따로 계산한다.
- DB 커넥션 풀은 QPS가 아니라 DB 점유 시간까지 포함해 계산한다.
- Kafka/Redis는 replication factor와 메모리 오버헤드를 포함한다.
- 산정 결과는 부하 테스트와 운영 metric으로 보정한다.
