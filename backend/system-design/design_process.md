# 설계 프로세스

요구사항을 실제 아키텍처로 변환할 때 사용할 수 있는 실무형 설계 흐름입니다.

---

## 0. 문제 정의

- 해결하려는 문제를 한 문장으로 정의
- 사용자/비즈니스 목표를 명확히 기록
- 성공 기준(Success Metric) 정의

예시:
- 목표: "주문 API의 피크 시간 오류율을 1% 미만으로 낮춘다"
- 지표: p95 latency, 에러율, 주문 성공률

---

## 1. 요구사항 분류

### 기능 요구사항 (FR)
- 시스템이 반드시 해야 하는 동작
- 예: 주문 생성, 결제 승인, 주문 조회

### 비기능 요구사항 (NFR)
- 품질 속성
- 예: 가용성 99.9%, p95 200ms 이하, 데이터 정합성 보장 수준

### 제약 조건
- 조직/기술/비용 제약
- 예: AWS만 사용, PostgreSQL 유지, 월 인프라 비용 상한

---

## 2. 범위 확정

- 이번 설계에서 다루는 범위(In scope)와 제외 범위(Out of scope)를 명시
- MVP 범위를 먼저 고정하고 확장 범위는 별도 분리

---

## 3. 용량 및 트래픽 추정

최소한 다음은 숫자로 기록합니다.

- DAU/MAU
- QPS (평균/피크)
- 데이터 저장량(일/월 증가량)
- 읽기/쓰기 비율
- 이벤트 크기, 네트워크 대역폭

추정값은 "가정"으로 명시하고, 이후 검증 계획과 함께 관리합니다.

---

## 4. 고수준 아키텍처

- 핵심 컴포넌트 정의 (API, DB, Cache, MQ, Worker)
- 데이터 흐름 (Write Path / Read Path)
- 장애 전파 경로와 격리 경계(Boundary)

체크 포인트:
- 단일 장애 지점(SPOF) 존재 여부
- 병목 지점(Hotspot) 예상
- 데이터 정합성 경계 설정

---

## 5. 데이터 모델 및 API 설계

### 데이터 모델
- 엔티티와 관계
- 인덱스 전략
- 샤딩/파티셔닝 기준

### API 설계
- 리소스 모델, 상태 코드, 에러 포맷
- 멱등성(Idempotency) 전략
- 버저닝 정책

---

## 6. 핵심 의사결정과 트레이드오프

중요 선택은 반드시 ADR로 남깁니다.

예:
- 동기 호출 vs 비동기 이벤트
- 강한 일관성 vs 최종 일관성
- RDB 단일 DB vs 샤딩/NoSQL

각 선택마다 "선택 이유", "버린 대안", "리스크"를 기록합니다.

---

## 7. 신뢰성/운영 설계

- 장애 대응: 재시도, 타임아웃, 서킷 브레이커, DLQ
- 관측성: 로그, 메트릭, 트레이싱, 알람
- 배포 전략: 롤링/블루그린/카나리
- 복구 전략: 백업, 복구 시간 목표(RTO), 데이터 손실 허용치(RPO)

---

## 8. 검증 및 리뷰

- 설계 리뷰 체크리스트 기반 리뷰 수행
- 부하 테스트/장애 주입 테스트 계획 수립
- 운영 전환 조건(Go-live criteria) 명시

---

## 최소 산출물 템플릿

- 문제 정의 1페이지
- 요구사항 목록(FR/NFR/제약)
- 아키텍처 다이어그램 1장
- 데이터 모델 초안
- API 명세 초안
- ADR 3개 이상
- 리스크/대응 계획
- 테스트/운영 계획

---

## 자주 발생하는 실패 패턴

- NFR 없이 기능만 설계
- 트래픽 추정 없이 기술 선택
- 장애 시나리오 누락
- 트레이드오프 기록(ADR) 부재
- 운영/관측성 설계 후순위 처리
