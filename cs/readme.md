### 스레드, 프로세스, 코어의 수는 많을수록 좋을까?
- 스레드, 프로세스, 코어의 수가 많을수록 시스템의 성능이 향상될수 있지만, 실제로는 그렇지 않을 확률이 높다.
#### 스레드가 많으면?
- 스레드가 지나치게 많으면 스레드당 컨텍스트 스위칭 비용이 증가하여 실제 효율이 떨어질수밖에 없고 많은 스레드가 동시에 실행될 경우에 메모리, 캐시, 락 등의 자원등의 이용에 데드락이 발생할 가능성이 높아진다.
#### 프로세스가 많으면?
- 프로세스는 각각의 독립적인 메모리공간을 사용하는데 많은 프로세스가 동시에 실행되면 메모리 사용량이 급격히 증가할 수 있다.
- 프로세스간 통신(IPC)가 있는 경우에 이는 스레드의 컨텍스트 스위치보다 비용이 증가한다. 따라서 이로 인해 시스템 성능저하가 야기될 수도 있다. 
#### 코어가 많으면?
- 병렬처리향상을 기대할 수 있으나, 이는 SW수준에서 멀티코어 환경에 최적회 되어 있어야 한다.만일 단일스레드 작업이 주를 이루는 경우 이러한 자원을 활용할 수 없다. 그리고 HW적으로 CPU비용, 전력소비, 발열 문제등이 나타날 수 있다.

### 멱등성
- 멱등성 : 동일한 연산을 여러번 요청(or 적용)하더라도 결과가 달라지지 않는 성질
- 대표적으로 멱등한 메서드: 
  - GET
  - HEAD: 클라이언트가 서버로부터 HTTP헤더 정보만 가져오도록 요청 (GET 메서드와 동일하지만 본문은 반환하지 않음)
  - PUT
  - DELETE
  - TRACE: 서버측에서 클라이언트 요청이 어떻게 처리되는지 확인하기 위해 서버가 클라이언트의 원래요청을 그대로 반환
    - 네트워크 디버깅, 애플리케이션 테스트용으로 활용
    - 클라이언트 요청이 프록시서버 or 게이트웨이를 거쳐가는 동안 변경되었는지 확인할 때
    - 네트워크워크 상의 데이터 경로와 요청/응답의 변형 여부를 디버깅할때
    - 잘못된 응답이 발생했을 때, 요청패킷이 정상적으로 서버에 도달했는지 확인 할 때
    - 일반적으로 보안상의 이유로 TRACE 요청은 제한되거나 비활성화된 경우가 많음. 요청이 누군가에 의해 가로채여, 민감한 정보가 노출될 위험.
  - OPTIONS: 서버로부터 어떤 HTTP메서드가 해당 리소스에 대해 허용되는지 메타데이터 형태로정보를 요청하는 메서드(요청헤드만 반환하며, 본문은 포함되지 않음)
    - 클라이언트와 서버 간의 **의사소통 가능 여부(호환성)** 확인할 때.
    - 특정 리소스가 지원하는 HTTP 메서드와 서버의 **CORS 정책**(Cross-Origin Resource Sharing, 교차 출처 요청 허용 여부)을 확인하기 위한 프리플라이트 요청(Preflight Request)으로 자주 사용 &rarr; 브라우저가 자동으로 options 메서드 호출
    - API 문서화 없이도 서버가 제공하는 기능을 탐색하고자 할 때.

### SSR, CSR
- SSR(Server Side Rendering)
  - MPA (Multi Page Application)
  - 서버 측에서 렌더링하는 방식
  - 클라이언트가 서버에 컨텐츠를 요청하면, 서버는 페이지에 필요한 데이터를 즉시 얻어와 모두 삽입하고, CSS까지 모두 적용해 렌더링 준비를 마친 HTML과 JS 코드를 응답 &rarr; 브라우저에서는 JS 코드를 다운로드하고, HTML에 JS를 연결.
  - 모든 데이터가 이미 HTML에 담긴 채로 브라우저에 전달되기 때문에 SEO에 유리
  - JS 코드를 다운로드 받고 실행하기 전에 사용자가 이미 렌더링된 HTML을 볼 수 있으므로, JS 다운로드를 기다려야 하는 CSR에 비해 초기 구동 속도가 빠름
- CSR(Client Side Rendering)
  - SPA (Single Page Application)
  - 클라이언트 측에서 렌더링하는 방식
  - 클라이언트가 서버에 컨텐츠를 요청하면, 서버는 빈 뼈대만 있는 HTML을 응답
  - 클라이언트는 연결된 JS 링크를 통해 서버로부터 다시 JS 파일을 다운로드 받은 뒤, JS를 통해 동적으로 페이지를 만들어 브라우저에 표현.
  - 단점
    1. 빈 뼈대만 있는 HTML을 받아오기 때문에 웹 크롤러 봇 입장에서 색인할만한 콘텐츠가 존재하지 않아 SEO에 불리
    2. 브라우저가 JS 파일을 다운로드하고, 동적으로 DOM을 생성하는 시간을 기다려야 하기 때문에 초기 로딩 속도가 느림. 
  - 장점
    1. 초기 로딩 이후 페이지 일부를 변경할 때에는 서버에 해당 데이터만 요청하면 되기 때문에 이후 구동 속도가 빠름. 
    2. 서버는 HTML 뼈대를 넘겨주는 역할만 수행하면 되므로 서버 측의 부하가 적음
    3. 클라이언트 측에서 연산과 라우팅 등을 직접 처리하기 때문에 반응속도가 빠르고 UX도 우수
