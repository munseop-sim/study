# 운영체제 핵심 개념

## CPU 스케줄링

### 정의

**CPU 스케줄링**은 운영체제가 프로세스들에게 공정하고 합리적으로 CPU 자원을 배분하는 것을 의미합니다. CPU 스케줄링이 없다면 반드시 실행되어야 할 프로세스가 실행되지 못하거나, 급하지 않은 프로세스가 실행되는 등 무질서한 상태가 발생할 수 있습니다.

### 선점형 vs 비선점형 스케줄링

#### 선점형 스케줄링(Preemptive Scheduling)

프로세스가 CPU를 사용하고 있더라도 운영체제가 프로세스로부터 자원을 **강제로 빼앗아** 다른 프로세스에 할당할 수 있는 방식입니다.

- **장점**: 응답 시간이 낮고 효율적
- **단점**: 컨텍스트 스위칭 오버헤드 발생, 경쟁 상태 발생 가능

#### 비선점형 스케줄링(Non-preemptive Scheduling)

하나의 프로세스가 자원을 사용하고 있다면 **해당 프로세스가 종료되거나 스스로 대기 상태로 전환**되기 전까지 다른 프로세스가 자원을 점유할 수 없는 방식입니다.

- **장점**: 컨텍스트 스위칭 비용이 상대적으로 적음
- **단점**: 응답 시간이 길 수 있음

### CPU 스케줄링 알고리즘

#### 1. FCFS(First Come First Served) - 선입 선처리

준비 큐에 삽입된 순서대로 프로세스를 처리하는 **비선점형** 방식입니다.

- **단점**: **호위 효과(Convoy Effect)** 발생 가능
  - 예: A(10ms), B(5ms), C(2ms) 순서일 때, C는 2ms 실행을 위해 15ms를 기다려야 함

#### 2. SJF(Shortest Job First) - 최단 작업 우선

CPU 이용 시간이 **가장 짧은 프로세스**부터 실행하는 **비선점형** 방식입니다.

- **장점**: 호위 효과 방지, 평균 대기 시간 단축
- **단점**: CPU 실행 시간이 긴 프로세스는 **기아 현상(Starvation)** 발생 가능

> **기아 현상(Starvation)**: 프로세스가 자원을 할당받지 못하고 무한정 준비 큐에 존재하게 되는 상황

#### 3. Round Robin(RR) - 라운드 로빈

선입 선처리에 **타임 슬라이스(Time Slice)** 개념이 추가된 **선점형** 방식입니다.

- 정해진 타임 슬라이스만큼의 시간 동안 돌아가며 CPU 이용
- **타임 슬라이스가 크면**: 선입 선출과 비슷해져 호위 효과 발생
- **타임 슬라이스가 작으면**: 컨텍스트 스위칭 비용이 커짐

#### 4. SRT(Shortest Remaining Time) - 최소 잔여 시간 우선

SJF에 선점형 스케줄링을 혼합한 방식입니다.

- 프로세스 실행 중 실행 시간이 더 짧은 프로세스가 추가되면 현재 프로세스를 중단하고 새 프로세스에 CPU 할당

#### 5. 다단계 큐 스케줄링(Multilevel Queue Scheduling)

**우선순위별로 준비 큐를 여러 개** 사용하는 방식입니다.

- 우선순위가 가장 높은 큐의 프로세스를 먼저 처리
- 큐별로 타임 슬라이스 지정 또는 다른 스케줄링 알고리즘 적용 가능
- **단점**: 기아 현상 발생 가능

#### 6. 다단계 피드백 큐 스케줄링(Multilevel Feedback Queue Scheduling)

다단계 큐 스케줄링에 **에이징(Aging) 기법**을 적용한 방식입니다.

- 새로 준비 상태가 된 프로세스는 가장 높은 우선순위 큐에 삽입
- 타임 슬라이스 동안 실행이 끝나지 않으면 다음 우선순위 큐로 이동 (CPU를 오래 사용해야 하는 프로세스는 우선순위 낮아짐)
- 낮은 우선순위 큐에서 너무 오래 기다린 프로세스는 높은 우선순위 큐로 이동 (에이징)

> **에이징(Aging)**: 오랫동안 대기한 프로세스의 우선순위를 점차 높이는 기아 현상 방지 방법

### 스케줄링 알고리즘 비교

| 알고리즘 | 유형 | 호위 효과 | 기아 현상 | 특징 |
|---------|------|-----------|-----------|------|
| FCFS | 비선점 | 발생 | 없음 | 단순, 공정 |
| SJF | 비선점 | 방지 | 발생 | 평균 대기 시간 최소 |
| Round Robin | 선점 | 방지 | 없음 | 타임 슬라이스 크기가 핵심 |
| SRT | 선점 | 방지 | 발생 | SJF의 선점형 버전 |
| 다단계 큐 | 선점 | 방지 | 발생 | 유형별 우선순위 관리 |
| 다단계 피드백 큐 | 선점 | 방지 | 방지 | 에이징으로 기아 현상 해결 |

## 시스템 콜(System Call)

### 배경: 자원 보호의 필요성

운영체제는 사용자가 실행하는 프로그램이 하드웨어 자원에 직접 접근하는 것을 방지하여 자원을 보호합니다.

- 프로그램이 CPU, 메모리, 하드 디스크에 마음대로 접근하면 자원이 무질서하게 관리될 수 있음
- 한 프로그램의 실수가 전체 컴퓨터에 영향을 줄 수 있음

### 이중 모드(Dual Mode)

CPU가 명령을 실행하는 모드를 **사용자 모드**와 **커널 모드**로 구분하는 방식입니다.

#### 사용자 모드(User Mode)

- 운영체제 서비스를 제공받을 수 없는 실행 모드
- 입출력 명령어와 같은 하드웨어 자원 접근 명령을 실행할 수 없음
- 일반 프로그램은 기본적으로 사용자 모드로 실행

#### 커널 모드(Kernel Mode)

- 운영체제 서비스를 제공받을 수 있는 실행 모드
- 자원에 접근하는 명령어를 비롯한 모든 명령어를 실행할 수 있음
- 운영체제는 커널 모드로 실행

### 시스템 콜(System Call) 정의

**시스템 콜**은 사용자 모드로 실행되는 프로그램이 운영체제 서비스(자원 접근 등)를 제공받기 위해 운영체제에 보내는 요청입니다.

- 일종의 **소프트웨어 인터럽트**
- 시스템 콜 명령어가 실행되면:
  1. CPU가 현재까지의 작업을 백업
  2. 커널 영역 내 시스템 콜을 수행하는 **인터럽트 서비스 루틴** 실행
  3. 기존 실행하고 있었던 프로그램으로 복귀하여 실행 계속

### 시스템 콜의 주요 유형

| 유형 | 예시 |
|------|------|
| 프로세스 제어 | fork(), exec(), exit(), wait() |
| 파일 관리 | open(), close(), read(), write() |
| 장치 관리 | ioctl(), read(), write() |
| 정보 유지 | getpid(), alarm(), sleep() |
| 통신 | pipe(), socket(), send(), recv() |

## 가상화(Virtualization)

### 정의

**가상화**란 하나의 물리적인 컴퓨팅 리소스를 논리적으로 분리하여 여러 개의 가상 리소스를 생성해 사용하는 기술입니다.

- 서버, 스토리지, 네트워크 등 다양한 IT 리소스를 가상화 가능
- 클라우드 컴퓨팅의 핵심 기술

### 가상화의 필요성

- **리소스 효율화**: 물리 서버에 여러 가상 서버를 배치하여 하드웨어 리소스를 효율적으로 사용
- **비용 절감**: 물리적인 하드웨어 수를 줄여 초기 구축 비용 및 유지 관리 비용 절감
- **격리성**: 하나의 가상 리소스에 장애가 발생해도 다른 가상 리소스에 영향 없음
- **유연성**: 가상 리소스를 필요에 따라 빠르게 생성하거나 삭제 가능
- **가용성**: 소프트웨어 기반이라 장애 복구가 용이하고 가용성 높음

### 서버 가상화

물리적인 서버를 여러 개의 **가상 머신(VM, Virtual Machine)**으로 나누어 사용하는 기술입니다.

- 하나의 물리 서버에서 여러 운영 체제를 실행 가능
- 각 VM은 독립적으로 운영 (VM = 게스트, 물리 서버 = 호스트)

#### 하이퍼바이저(Hypervisor)

물리 서버에 설치되는 가상화 소프트웨어입니다. 하드웨어로부터 VM에 필요한 CPU, 메모리 등의 리소스를 할당하고 VM들이 서로 격리되어 동작하도록 관리합니다.

| 유형 | 특징 | 예시 |
|------|------|------|
| **Type 1** (Native/Bare Metal) | 하드웨어에서 직접 실행, 별도 호스트 OS 불필요, 엔터프라이즈 데이터 센터에서 사용 | KVM, Microsoft Hyper-V |
| **Type 2** (Hosted) | 호스트 OS 위에서 하나의 애플리케이션으로 실행, 개인용 PC/개발 환경에서 주로 사용 | Oracle VirtualBox, VMware Workstation |

### VM vs 컨테이너 비교

| 구분 | VM | 컨테이너 |
|------|-----|---------|
| 기술 | 하이퍼바이저를 통해 게스트 OS 실행 | 호스트 OS의 커널 공유 |
| 격리 수준 | 높음 | 상대적으로 낮음 |
| 무게 | 무거움 (게스트 OS 포함) | 가벼움 |
| 성능 | 느림 | 빠름 |
| 확장성 | 낮음 | 높음 |
| 보안 | 강함 | 커널 공유로 취약할 수 있음 |

> VM과 컨테이너는 서로 대체하는 기술이 아니며, VM 위에서 컨테이너를 실행하는 방식과 같이 목적에 따라 함께 활용 가능합니다.

## 페이지 교체 알고리즘(Page Replacement Algorithm)

### 배경: 가상 메모리와 요구 페이징

**가상 메모리 관리 기법(Virtual Memory Management)**은 프로그램의 일부만을 메모리에 적재하여 실제 사용 가능한 물리 메모리 양보다 큰 프로세스를 실행할 수 있도록 하는 기법입니다.

**페이징**: 메모리의 물리 주소 공간을 프레임 단위로 나누고, 프로세스의 논리 주소 공간을 페이지로 나누어 프레임에 할당하는 방식

**요구 페이징(Demand Paging)**: 실제로 필요할 때만 페이지를 메모리에 적재하는 방식

새로운 페이지를 할당하려고 하는데 공간이 부족한 경우, 메모리에 존재하는 다른 페이지와 신규 페이지를 교체해야 합니다. 이때 교체 대상을 결정하는 방법이 **페이지 교체 알고리즘**입니다.

### 페이지 교체 알고리즘 종류

#### 1. FIFO(First In First Out)

**적재된 순서대로** 페이지를 교체하는 알고리즘입니다.

- 먼저 적재되었다는 이유만으로 교체되는 비효율 존재
- 변형 알고리즘: **2차 기회 페이지 교체 알고리즘**

#### 2. OPT(Optimal Page Replacement) - 최적 페이지 교체

**앞으로 가장 나중에 사용될 페이지**를 교체합니다.

- 페이지의 미래 사용 빈도와 접근 패턴을 알기 어렵기 때문에 **이론적 알고리즘**으로 분류
- 다른 알고리즘의 성능 비교 기준으로 사용

#### 3. LRU(Least Recently Used)

**가장 오래 사용되지 않은 페이지**를 교체합니다.

- 과거 사용 이력을 기반으로 미래 접근 패턴 예측
- **최적 근접 알고리즘**으로 분류
- 단, 페이지별 마지막 접근 시간 관리가 필요하여 메모리 오버헤드 발생

#### 4. LFU(Least Frequently Used)

**가장 적게 사용된 페이지**를 교체합니다.

- 사용 빈도가 같은 경우 다른 알고리즘 사용 가능
- **최적 근접 알고리즘**으로 분류
- 페이지별 사용 횟수 관리가 필요하여 메모리 오버헤드 발생

#### 5. NUR(Not Used Recently)

**최근에 사용된 적이 없는 페이지**를 교체합니다.

- LRU, LFU와 유사한 성능이면서 **메모리 오버헤드를 줄인** 방식
- 페이지 테이블 엔트리의 **참조 비트**와 **수정 비트**를 활용하여 교체 대상 결정
- **최적 근접 알고리즘**으로 분류

### 알고리즘 비교

| 알고리즘 | 교체 기준 | 구현 복잡도 | 성능 | 분류 |
|---------|---------|------------|------|------|
| FIFO | 적재 순서 | 낮음 | 낮음 | - |
| OPT | 미래 사용 시점 | 불가(이론적) | 최적 | 이론적 알고리즘 |
| LRU | 마지막 접근 시간 | 보통 | 높음 | 최적 근접 |
| LFU | 사용 빈도 | 보통 | 높음 | 최적 근접 |
| NUR | 참조/수정 비트 | 낮음 | LRU와 유사 | 최적 근접 |

## 참조 지역성의 원리(Locality of Reference)

### 정의

**참조 지역성의 원리**는 CPU가 메모리에 접근할 때 주된 경향을 바탕으로 만들어진 원리입니다.

- 주로 **캐시 메모리의 적중률을 높여** CPU의 메모리 접근 횟수를 줄이는 데 이용

### 두 가지 지역성

#### 시간 지역성(Temporal Locality)

CPU는 **최근에 접근했던 메모리 공간에 다시 접근**하려는 경향이 있습니다.

- 예: 지역 변수에 값을 저장하면 나중에 다시 지역 변수에 접근할 가능성이 높음
- 예: 반복문(loop) 내의 변수는 반복적으로 접근됨

#### 공간 지역성(Spatial Locality)

CPU는 **접근한 메모리 공간 근처에 접근**하려는 경향이 있습니다.

- 예: CPU가 인텔리제이 프로그램을 실행할 때 인텔리제이 프로그램이 모여 있는 공간 근처를 집중적으로 접근
- 예: 배열 요소를 순차적으로 접근하면 연속된 메모리 영역에 접근

### 실무 예시: 2차원 배열 순회 최적화

```java
// 비효율적 순회 (열 우선) - 약 577ms
for (int i = 0; i < size; i++) {
    for (int j = 0; j < size; j++) {
        array[j][i]++;  // 공간 지역성 위반: 연속되지 않은 메모리 접근
    }
}

// 효율적 순회 (행 우선) - 약 28ms
for (int i = 0; i < size; i++) {
    for (int j = 0; j < size; j++) {
        array[i][j]++;  // 공간 지역성 활용: 연속된 메모리 접근
    }
}
```

Java에서 2차원 배열은 내부적으로 1차원 배열에 대한 참조 배열입니다. `array[i]`는 각각 독립된 `int[]` 객체로 메모리상 반드시 연속적으로 배치되는 것이 보장되지 않습니다.

**열 우선 순회**는 캐시 히트율이 낮아지고, **행 우선 순회**는 CPU 캐시의 공간 지역성을 활용하여 캐시 히트율이 높아집니다.

### 캐시와의 관계

CPU 캐시는 공간 지역성 원리에 근거하여 **물리적인 메모리 공간상에 인접한 데이터를 미리 캐싱**합니다. 참조 지역성 원리를 잘 활용하면:
- 캐시 적중률(Cache Hit Rate) 향상
- 메인 메모리 접근 횟수 감소
- 전체 프로그램 성능 향상

## 명령어 파이프라이닝(Instruction Pipelining)

### 정의

**명령어 파이프라이닝**은 CPU가 여러 명령어를 동시에 처리하기 위해 각 명령어를 여러 단계로 분할하고, 각 단계를 다른 명령어와 겹쳐서 실행하는 방법입니다.

- 전통적인 CPU: 한 번에 하나의 명령어를 처리
- 파이프라인 CPU: 여러 명령어를 각기 다른 단계에서 동시에 처리

### 파이프라인 단계 예시

| 단계 | 설명 |
|------|------|
| IF (Instruction Fetch) | 명령어 가져오기 |
| ID (Instruction Decode) | 명령어 해석 |
| EX (Execute) | 명령어 실행 |
| MEM (Memory Access) | 메모리 접근 |
| WB (Write Back) | 결과 저장 |

하나의 명령어가 EX 단계에 있을 때, 다음 명령어는 ID, 그 다음 명령어는 IF 단계를 동시에 수행합니다.

### 파이프라인 위험(Pipeline Hazard)

파이프라이닝이 성능 향상에 실패하는 경우를 **파이프라인 위험**이라고 합니다.

#### 1. 데이터 위험(Data Hazard)

**데이터 의존적인 두 명령어를 동시에 실행**하는 경우 파이프라인이 제대로 작동하지 않는 것입니다.

- 예: 명령어 B가 명령어 A의 연산 결과 R에 의존하는 경우, 명령어 A의 결과 저장 단계가 완료되어야 명령어 B가 R을 사용 가능

#### 2. 제어 위험(Control Hazard)

**분기(branch)나 조건문** 등으로 인해 다음에 실행할 명령어가 무엇인지 결정되지 않아 파이프라인이 멈추게 되는 것입니다.

- 분기 예측 기법이 사용되기도 하지만, 예측 실패 시 계산된 파이프라인을 모두 버려야 하여 성능 저하 발생

#### 3. 구조적 위험(Structural Hazard)

한 명령어가 자원을 사용하면 **해당 자원이 필요한 다른 명령어는 멈추게** 되어 파이프라인이 제대로 작동하지 않는 것입니다.

- 서로 다른 명령어가 동시에 ALU, 레지스터와 같은 CPU 자원을 사용하려 할 때 발생

## 디스크 접근 시간(Disk Access Time)

### 정의

특정 데이터 블록을 읽거나 쓰기 위해 필요한 모든 과정을 수행하는 데 걸리는 시간입니다.

```
디스크 접근 시간 = 탐색 시간 + 회전 지연 시간 + 데이터 전송 시간
```

### 구성 요소

#### 1. 탐색 시간(Seek Time)

**헤드를 데이터가 존재하는 트랙으로 이동**시키는 데 소요되는 시간입니다.

- 기계적인 이동이기 때문에 시동 시간이 필요
- 액세스할 데이터들의 **지역성이 높은 경우** 헤드의 이동 거리가 짧아져 평균 탐색 시간 단축 가능

#### 2. 회전 지연 시간(Rotational Latency)

**원하는 데이터가 저장된 섹터가 헤드 아래로 회전**되어 올 때까지 기다리는 데 소요되는 시간입니다.

#### 3. 데이터 전송 시간(Data Transfer Time)

**데이터를 전송**하는 데 소요되는 시간입니다.

- 블록의 크기, 회전 속도, 트랙의 저장 밀도, 버스 전송률 및 제어기 내부 전자회로의 속도에 따라 결정

### 랜덤 액세스 vs 순차 액세스

#### 랜덤 액세스가 느린 이유

디스크의 다른 위치에 흩어져 있는 데이터 블록을 읽어야 하기 때문에:
- 헤드를 여러 번 움직여야 함 (탐색 시간 증가)
- 각 블록에 접근할 때마다 회전 지연 시간을 기다려야 함
- 기계적인 이동으로 인한 지연이 누적됨

#### 순차 액세스가 빠른 이유

데이터가 연속적으로 저장되어 있어:
- 헤드 이동이 상대적으로 적음 (탐색 시간 최소화)
- 회전 지연 없이 연속적으로 데이터 읽기 가능

> SSD의 경우 기계적 부품이 없어 랜덤 액세스와 순차 액세스의 성능 차이가 HDD에 비해 훨씬 적습니다.
