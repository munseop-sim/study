# 클래스 다루기

## 클래스의 토대: 추상데이터형(ADT, Abstract Data Type)
### 이점
- 구현 세부사항을 감출 수 있다.
- 변경이 전체에 영향을 미치지 않는다.
- 인터페이스가 더 많은 정보를 제공하도록 만들 수 있다.
- 성능을 향상시키기 쉽다.
- 프로그램이 명확해진다.
- 프로그램의 가독성이 높아진다.
- 전체프로그램에 데이터를 넘길 필요가 없다.
- 저수준 구현구조체 대신 현실세계의 개체를 다룰 수 있다.

### 원칙
- 전형적인 저수준 데이터형을 저수준 데이터형이 아닌 ADT로 만들거나 사용한다.
  - 가능한 높은 추상화수준에서 이해하려고 하자
- 파일과 같은 일반적인 객체를 ADT로 취급한다.
  - OS의 File과 같이 복잡한 세부사항에 신경쓰지 않도록 디스크공간을 "파일"로 처리할 수 있는 것처럼 현실세계를 반영하는 ADT계층을 높은 수준으로 구현하자
- 간단한 객체도 ADT로 취급한다.
- ADT가 저장된 매체와 독립적으로 ADT를 참조한다.
### ADT + 상속 + 다형성 = 클래스

## 좋은 클래스 인터페이스
### 좋은 추상화 : 복잡한 연산을 단순한 형태로 보여주는 능력
- 클래스 인터페이스가 일관된 추상화 수준을 갖도록 한다.
- 각 클래스는 오직 하나의 ADT만 구현해야 한다. 
  - 추상화 수준이 뒤죽박죽일때에는 프로그램을 점점 더 이해하기 어렵고 유지보수 할 수 없게 된다.
- 클래슥 구현하고 있는 추상화가 무엇인지 이해해야 한다.
- 서로 반대되는 기능을 갖는 서비스 쌍을 제공해야 한다.
- 관련이 없는 정보를 다른클래스로 옮겨야 한다.
- 가능하면 인터페이스를 의미론적이기 보다는 프로그래밍적으로 만들어야 한다.
- 코드변경시에 인터페이스의 추상화가 망가지지 않도록 해야 한다.
- 인터페이스 추상화에 맞지 않는 public 멤버를 추가하지 말자.
- 추상화와 응집도를 함께 고려하자.
### 좋은 캡슐화
- 추상화 : 구현세부사항을 무시할 수 있는 모델을 제공함으로써 복잡성 관히에 도움을 준다.
- 클래스와 멤버의 접근을 최소화한다.
    >어떻게 해야  인터페이스 추상화의 무결성을 최상으로 유지할 수 있는가?
  - 확신이 없다면 private로 선언한다.
- 멤버데이터를 public으로 노출하지 말자
- 내부 구현 세부사항을 클래스의 인터페이스에 입력하지 말자.
- 클래스의 사용자를 가정하지 말라.
  - 인터페이스에 적혀있는것 외에는 인터페이스가 어떻게 사용될 것인지 또는 어떻게 사용되지 않읗 것인지에 대해서 어떠한 가정도 하지 말자.
- friend 클래스는 사용하지 말자.
    - friend 클래스는 캡슐화를 깨뜨린다.
- 코드를 작성할 때 편의성보다는 가독성이 높은 코드를 작성하자.
- 캡슐화의 의미론적인 위반을 각별히 주의하자.
- 지나치게 밀접한 결합은 주의하자.
