# `Layered`
- 공통적으로 사용되는 소프트웨어 개발 방식 중 하나로, 시스템을 여러 계층으로 분할하는 방법을 의미
- 각 계층은 특정 기능 집합을 처리하며, 일반적으로 계층들은 구체성에 따라 정렬
- 예를 들어, UI 계층, 비즈니스 로직 계층, 데이터 엑세스 계층 등을 사용할 수 있음
- 장점
  1. 재사용성: 각 계층이 정의된 기능에 집중하기 때문에, 상위 계층은 하위 계층에 정의된 기능을 작업에 필요에 따라 재사용 가능
  2. 유지관리: 컴포넌트가 논리적으로 분리되어 있기 때문에, 각 계층의 유지 보수와 수정이 용이
  3. 변화에 대한 유연성: 한 계층의 변화가 다른 계층에 직접적인 영향을 주지 않기 때문에, 시스템 전체를 변경하지 않고도 한 계층의 기술이나 구현을 변경할 수 있음
  4. 개발의 단순성: 각 계층은 다른 계층의 결과에 의존하기 때문에, 개발자는 특정 계층에 집중하며 작업을 수행가능
- 단점
  1. 성능: 하위 계층을 반복적으로 호출함으로써 발생하는 오버헤드가 성능에 영향을 줄 수 있음.
  2. 복잡성: 때로는 실제로 필요하지 않은 레이어를 포함하는 경우가 있을 수 있음으로써 설계와 구현이 복잡해질 수 있다.
  3. **묶여 있는 결합도**: 한 계층이 다음 계층에 너무 의존하게 되면, 계층간의 결합도가 증가할 수 있습니다. 이로 인해, 중간 계층에 발생하는 문제가 상위 계층에 영향을 줄 수 있음

# `HEXAGONAL`
- 헥사고날 아키텍처(또는 포트와 어댑터 아키텍처)는 소프트웨어 설계 패턴 중 하나
- 내부(Service, 도메인 로직): 도메인 로직이 위치하며, `in.port`를 구현하고 `out.port`를 주입받아서 사용한다.  
- 외부(Adapter):  외부로부터 도메인 로직에 접근 또는 도메인 로직에서 외부로 접근하는 자원을 의미
- 인터페이스(Port): 도메인 로직-외부간의 연결
- 장점
  1. **추상화**: 핵심 비즈니스 로직은 외부 구현 세부사항과 분리
  2. **테스트 용이성**: 핵심 비즈니스 로직은 고립된 상태에서 쉽게 테스트할 수 있음
  3. **유연성**: 다양한 인프라스트럭처 기술을 쉽게 사용하거나 교체할 수 있습니다. 이는 시스템의 변경 사항에 대한 대응력을 향상
  4. **장기적인 관리**: 기술적인 부분과 비즈니스 로직이 분리되어 있어, 애플리케이션의 유지보수가 용이함
- 단점
  1. **복잡도**: 처음에는 복잡한 아키텍처가 과도한 것으로 나타날 수 있습니다. 작은 프로젝트 또는 간단한 애플리케이션에 대해서는 오버 엔지니어링일 수 있음.
  2. **학습곡선**: 개발자들이 이 아키텍처를 이해하고 효과적으로 사용하기 위해서는 시간이 필요할 수 있음.
